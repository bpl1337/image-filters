# Отчёт: фильтрация изображений на CPU и GPU

**Условия:** Windows x64, Release-сборка (`g++ -O3 -march=native -mavx2 -fopenmp`), MSYS2 UCRT64 GCC, Intel iGPU через OpenCL. Среднее по 10 повторам с прогревом.

Стиль реализации — функциональный: фильтры — чистые функции `ImageView → ImageView`, immutable входы, `std::transform`/лямбды, бенчмарк через higher-order `MeasureMilliseconds`.

## Зависимость от размера изображения

Размер исходного и результирующего изображения у нас одинаковый — все три фильтра не меняют разрешение картинки. Поэтому интерес представляет одна величина — число пикселей N.

Теоретически у всех фильтров сложность O(N): каждый пиксель обрабатывается независимо за константное число операций. На графике "время vs N" это даёт прямую линию, проходящую через начало координат.

На практике для маленьких размеров видны константные накладные расходы (~0.3 мс на запуск OpenMP-пула или копирование на GPU). Из-за этого кривые на малых N задраны вверх, но к большим N выходят на правильный линейный тренд.

### Поведение по фильтрам

**Инверсия — memory-bound.** Работа на пиксель минимальная (XOR с 0xFF). Узкое место — пропускная способность памяти, а не процессор. Поэтому параллелизм почти не помогает: все 4 реализации на 5.76 Мп дают примерно одно и то же время (5–11 мс). OpenCL даже проигрывает Sequential из-за копирования через PCIe.

Вывод: для лёгких per-pixel операций параллелить нет смысла — упираешься в DRAM.

**Медиана 3×3 — compute-bound.** Sorting network из ~25 сравнений на канал × 3 канала = ~75 операций на пиксель. Вычислений много, и они отлично распараллеливаются. На 5.76 Мп: Sequential 1145 мс, OpenMP 110 мс (×10.4), SIMD 59 мс (×19.4), OpenCL 9.8 мс (×116).

Вывод: на тяжёлых compute-bound задачах GPU даёт огромный выигрыш — на 2 порядка быстрее scalar.

**Sobel — средний случай.** 10 операций на пиксель + работа с одним каналом яркости. Ускорения умеренные: OpenMP ×2.2, SIMD ×2.5, OpenCL ×3.6. OpenCL обгоняет SIMD только на больших картинках (точка перелома ~1 Мп) — на меньших накладные расходы PCIe съедают выигрыш.

## Выводы

1. **Линейная сложность O(N)** подтверждается всеми 12 кривыми: время растёт пропорционально площади картинки. Например для Sequential-медианы: размер вырос в 6.4 раза (с 0.9 Мп до 5.76 Мп) — время выросло в 5.78 раза.

2. **Параллелизм имеет смысл только при достаточном объёме работы.** Если фильтр сам по себе занимает < 1 мс — OpenMP и OpenCL только замедляют. SIMD безопасен всегда (нет накладных расходов).

3. **Выбор backend'а зависит от характера задачи:**
   - Memory-bound (инверсия) → Sequential или SIMD одинаково хороши.
   - Compute-bound (медиана) → OpenCL вне конкуренции.
   - Средний случай (Sobel) → SIMD для малых картинок, OpenCL для больших.

4. **GPU имеет "стоимость входа" ~0.3–0.6 мс** на копирование данных через PCIe. На задачах быстрее этого порога — GPU всегда проиграет CPU-реализациям.
