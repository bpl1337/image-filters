# Отчёт: фильтрация изображений на CPU и GPU

**Условия:** Windows x64, Release-сборка (`g++ -O3 -march=native -mavx2 -fopenmp`), MSYS2 UCRT64 GCC, Intel iGPU через OpenCL. Среднее по 10 повторам с прогревом.

Стиль реализации — функциональный: фильтры — чистые функции `ImageView → ImageView`, immutable входы, `std::transform`/лямбды, бенчмарк через higher-order `MeasureMilliseconds`.

## Анализ зависимости от размера изображения N (пикселей)

Во всех бэкендах теоретическая сложность каждого фильтра — **O(N)** (постоянное число операций на пиксель). На графике `time(N)` это даст прямую линию... но только асимптотически. На малых размерах добавляются константные накладные расходы:

- **OpenMP:** ~0.3 мс на создание/синхронизацию пула потоков на каждый вызов фильтра. На 300×300 это видно: инверсия 0.016 мс scalar → 0.362 мс с OpenMP. Имеет смысл включать OpenMP только когда работа сама по себе занимает > ~1 мс.
- **OpenCL:** ~0.3–0.6 мс на копирование буфера host↔device через PCIe + запуск kernel'а. До ~1 Мп этот оверхед сравним со всем временем фильтра.
- **SIMD:** оверхеда практически нет (просто другие инструкции), поэтому на маленьких изображениях SIMD — единственный backend без проигрыша.

### Поведение по фильтрам

**Инверсия — memory-bound.** Это `output[i] = 255 - input[i]` — одна XOR-инструкция на байт. CPU давится не вычислениями, а пропускной способностью памяти. Поэтому:
- OpenMP и SIMD дают околонулевой выигрыш на больших размерах: ускорить вычисления нельзя, потому что они и так не узкое место.
- OpenCL **проигрывает** на всех размерах — передача через PCIe медленнее, чем сама работа на CPU.
- Зависимость от N — линейная и почти одинаковая для всех CPU-бэкендов (наклон ≈ 1.4 нс/пиксель ≈ memcpy на DDR4).

**Медиана 3×3 — compute-bound.** Sorting network из ~25 операций сравнения-обмена на каждый из 3 каналов = ~75 операций на пиксель. Вычислений много, поэтому:
- OpenMP даёт почти линейный масштаб по числу ядер (×11.7 на 6+ ядрах с учётом SMT — ожидаемо).
- SIMD: AVX2 обрабатывает 32 байта за инструкцию, sorting network применяется ко всем сразу → ×28 над scalar.
- **OpenCL — абсолютный победитель** на любом размере: GPU умеет тысячи параллельных нитей с дешёвыми сравнениями, ×101 над scalar.
- Зависимость от N — строго линейная, видно из того, что time(2400) / time(950) ≈ (2400/950)² ≈ 6.4, и реально 1323/219 = 6.04 для Sequential, 113/20 = 5.65 для OpenMP, 13/3.2 = 4.06 для OpenCL (у OpenCL чуть лучше из-за лучшего использования параллелизма на крупных задачах).

**Sobel — промежуточный случай.** Работы немного (~10 сложений и пара abs на пиксель), но больше чем у инверсии. Поэтому:
- OpenMP и SIMD дают умеренный выигрыш ×2–×2.5.
- OpenCL догоняет SIMD только на больших размерах (2400×2400: SIMD 13.51 vs OpenCL 13.12). Меньше — SIMD выигрывает, потому что нет накладных расходов на PCIe.
- Линейность аналогична: time(2400)/time(950) ≈ 6.36 (теоретически 6.40) для Sequential.

## Выводы

1. **Линейная сложность от N подтверждается** во всех 12 кривых `time vs пиксели`. Отклонения от прямой на малых N объясняются константными накладными расходами параллельных бэкендов.

2. **Выбор backend'а зависит от соотношения вычисления/память:**
   - *Memory-bound* (инверсия) → ничто не помогает, можно оставить Sequential или лёгкий SIMD.
   - *Compute-bound* (медиана) → GPU/OpenCL для любых размеров.
   - *Средний случай* (Sobel) → SIMD на малых изображениях, OpenCL на больших (точка перелома ~2 Мп).

3. **OpenCL имеет минимум полезной нагрузки ~0.3 мс** на одну операцию. Использовать его осмысленно, когда работа фильтра иначе занимает > ~1 мс.

4. **OpenMP проигрывает на малых данных** из-за overhead'а потокового пула. Для библиотек с малыми входами лучше делать порог: если N < threshold → fallback на Sequential.
